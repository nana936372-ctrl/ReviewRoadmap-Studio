# ReviewRoadmap Studio 产品设计文档

## 概述

ReviewRoadmap Studio 是一款面向独立 App 开发者和早期小团队的 AI 产品决策助手。第一版 demo 聚焦分析公开的 iOS App Store 评论，把零散用户声音转化为有证据支撑的产品路线图决策。

这个作品集项目的目标，是展示 AI 产品经理所需要的综合能力：用户理解、结构化产品思维、技术理解力，以及清晰克制的信息设计审美。

## 目标用户

主要用户：已经有上线 App、但缺少产品研究能力的独立开发者或小型 App 团队。

典型处境：

- 他们持续收到 App Store 评论，但没有足够时间逐条阅读和归纳。
- 他们很难快速判断评论里哪些是 bug、哪些是功能需求、哪些是新手引导问题、哪些是价格/付费阻力，哪些可能影响留存。
- 他们需要一个轻量工具，帮助自己判断下一步应该修什么、优化什么、探索什么。

作品集里的第二层用户：AIPM 面试官。这个产品需要让面试官看出，你能够把模糊的用户信号转化成清晰的产品决策。

## 产品定位

ReviewRoadmap Studio 不是一个通用评论总结器，而是一个“从用户证据到产品优先级”的决策工具。

一句话定位：

> 几分钟内，把 App Store 评论转化为有证据支撑的产品路线图。

产品体验应该更像一个聚焦的 PM 工作台，而不是聊天机器人。AI 的价值通过结构化输出、证据引用、优先级解释和实验建议体现出来。

## MVP 范围

第一版 MVP 支持一条完整 demo 闭环：

1. 用户输入 App Store 应用链接，或上传一组评论样例。
2. 系统把原始评论整理成干净的评论表。
3. AI 将评论内容分类为不同主题：
   - Bug 反馈
   - 功能请求
   - 新手引导阻力
   - 价格或付费墙阻力
   - 留存风险
   - 用户喜爱点
4. AI 将相似评论聚类为洞察组。
5. 系统生成三张路线图卡片：
   - Fix：紧急的稳定性或可用性问题
   - Improve：高确定性的体验优化
   - Explore：有潜力但仍需验证的机会点
6. 每张路线图卡片包含：
   - 产品建议
   - 优先级分数
   - 证据引用
   - 用户场景或用户分群
   - 建议关注的产品指标
   - 验证实验
   - 风险或反向证据
7. 用户可以导出一页产品决策 brief。

第一版 MVP 不做：

- 账号系统
- 团队协作
- 全量历史评论同步
- 自动竞品监控
- App Store Connect、Google Play、Chrome Web Store、G2、Zendesk 等原生集成
- 完整 PRD 生成

## Demo 数据策略

第一版 demo 使用 iOS App Store 评论数据，因为它公开、容易理解，也比 Chrome Web Store 或 B2B SaaS 评论更容易清洗。

为了保证面试或展示时稳定，demo 可以先使用一份准备好的样例数据集。产品界面仍然可以呈现为“输入 App Store 链接”，随后展示已导入的样例评论。

推荐 demo 品类：

- AI 写作类 App
- 习惯养成或效率类 App
- 语言学习类 App
- 笔记或知识管理类 App

最推荐优先选择 AI 写作或效率类 App，因为它和“个人 AI 产品创业者/独立开发者”的叙事更贴合。

## 核心页面

### 1. 输入页

目标：让用户觉得流程具体、轻量、低成本。

主要元素：

- App Store 链接输入框
- 可选评论样例上传
- App 品类选择
- 时间范围选择
- “Analyze Reviews” 主按钮

成功标准：

- 用户能立刻理解需要输入什么数据。
- 页面感觉像一个产品工具，而不是一个 prompt 输入框。

### 2. 评论智能分析仪表盘

目标：在输出路线图之前，先展示 AI 是如何理解评论的，让用户愿意相信后续结论。

主要元素：

- 评论数量和评分概览
- 主题分布
- 不同主题下的情绪倾向
- 带原文引用的证据表
- 高频问题和需求的聚类视图

成功标准：

- 用户能看到结论来自哪里。
- 界面既有审美，也能承载较高信息密度。

### 3. 路线图卡片

目标：把分析结果转化为产品决策。

主要元素：

- 三张卡片：Fix、Improve、Explore
- 带解释的优先级分数
- 用户评论证据引用
- 指标假设
- 建议验证实验
- “为什么不是现在做”或风险说明

成功标准：

- 输出更像产品判断，而不是普通总结。
- 每条建议都能追溯到原始用户证据。

### 4. 产品决策 Brief 导出页

目标：形成一个适合作品集展示的最终产物。

主要元素：

- 问题总结
- 关键用户证据
- 推荐路线图
- 实验计划
- 尚待验证的问题

成功标准：

- 导出的内容像 PM 会发给 founder 或团队的一页产品判断。
- 视觉上能展示结构化思考和信息设计审美。

## AI 工作流

AI 工作流需要分阶段设计，让输出可解释、可追溯：

1. 评论标准化：
   - 提取评分、标题、正文、日期、版本号等字段。
   - 去除重复评论和空评论。
   - 保留原始评论文本，用于后续证据引用。
2. 信号分类：
   - 为每条评论打上一个或多个标签。
   - 识别情绪倾向和紧急程度。
   - 推断可能的用户意图或使用场景。
3. 洞察聚类：
   - 将语义相似的评论分组。
   - 用产品语言为每个聚类命名。
   - 绑定代表性证据。
4. 机会点评分：
   - 估算频次、严重程度、业务影响、信心和实现成本。
   - 评分要透明，不要伪装成过度精确的数学模型。
5. 生成路线图卡片：
   - 将高价值机会点转化为可执行的产品建议。
   - 包含指标和验证实验。
6. 生成决策 brief：
   - 用清晰、精炼、有层次的方式总结完整判断链路。

产品不应该假装 AI 知道超过数据本身的信息。当信心不足时，要明确说明，并建议进一步验证。

## 视觉方向

界面应该像一个清爽、现代、有判断力的 PM 决策工作台：

- 信息密度高，但整体安静克制
- 信息层级清晰
- 字体层级有明确主次
- 尽量少用装饰性元素
- 用数据卡片、证据引用和决策卡片形成主要视觉记忆点
- 色彩克制，但用有意义的强调色区分不同信号类型

最有审美记忆点的瞬间，应该是从“杂乱评论证据”过渡到“清晰路线图卡片”的过程。

## 作品集叙事

在 AIPM 作品集中，可以这样讲这个项目：

1. 问题：小型 App 团队会收集到很多用户反馈，但很难把这些反馈转化为产品决策。
2. 洞察：AI 不应该只总结反馈，而应该保留证据，并帮助 PM 做取舍。
3. 产品：ReviewRoadmap Studio 将 App Store 评论转化为洞察聚类和路线图卡片。
4. AI 设计：工作流拆成分类、聚类、评分、建议生成和证据引用，避免黑箱式输出。
5. PM 判断：产品强调做什么、不做什么、关注什么指标，以及下一步如何验证。
6. 结果：demo 能同时展示产品审美、用户同理心、技术结构和决策质量。

## 成功标准

这个作品集产品成功的标准是：面试官能在三分钟内理解以下几点：

- 产品服务谁
- 问题为什么真实存在
- AI 的价值为什么不只是总结
- 系统如何支持产品决策
- 为什么创作者同时具备产品审美和技术理解力

demo 至少要产出一条有说服力的路线图建议，并且这条建议能追溯回原始用户评论证据。

## 风险与应对

风险：产品看起来像普通 ChatGPT 套壳。

应对：使用结构化仪表盘、证据引用、评分因素和路线图卡片，不把主体验做成开放式聊天。

风险：评论抓取或实时导入不稳定。

应对：demo 使用准备好的样例数据集，将实时导入作为后续可扩展集成。

风险：AI 建议显得过度自信。

应对：加入置信度、反向证据和验证实验。

风险：产品范围变得过大。

应对：MVP 只聚焦 iOS App Store 评论和一个核心输出：有证据支撑的路线图卡片。

## 实施备注

第一版可以做成一个精致的 Web App 原型，使用本地样例数据。相比后端完整性，优先保证端到端 demo 的清晰度和完成度。

建议技术栈：

- 前端：Next.js 或 Vite React
- 样式：Tailwind CSS 或项目已有设计系统
- AI 层：分阶段 prompt 或结构化输出
- 数据：第一版使用静态 JSON 样例评论

实现时要让样例数据层和真实导入层保持可替换，方便后续接入真实数据源。
